# Issue #39 — fix(reader): page-flip animation does not visually run

- **Status**: Resolved
- **Milestone**: M6 — Reader Enhancements
- **Branch**: `fix/39-page-flip-motion`

## Symptom

When advancing a page in the reader — by click, Next button, ArrowRight, scrubber, or page-jump — no visible flip motion played. The new page simply snapped in. The bug was preset-agnostic (Classic, Curl, Slide, Fade, Book Tilt all snapped) and mode-agnostic (single and spread).

## Root Cause

`PageFlip` mounted the outgoing leaf with its **final** style already applied in the first render: `opacity: 0`, `transform: rotateY(-180deg)`, etc. A CSS `transition` interpolates between two states — but the element was born in its destination state, so the browser had no "from" frame to animate from. The transition declaration was present in `getComputedStyle` (so the previous reduced-motion E2E still passed), but no frame between the resting and final state was ever painted.

The original implementation from #8 had the same defect; it only *felt* animated because the static page kept dimming via the `outgoing` flag, masking the missing flip. The refactor in #36 made the bug obvious because Slide and Tilt have no static-page dim companion.

## Fix

- **Preset registry now exposes two frames**, not one. `FlipPreset.build(direction, duration)` returns `FlipFrames = { initial: CSSProperties; final: CSSProperties }`. The `initial` frame represents the resting state (opacity 1, identity transform, same transition string as `final`); the `final` frame is the old single-style. Every preset was updated:
  - **Classic** — initial `transform: none`, final `rotateY(±180deg)`.
  - **Curl** — initial `transform: none` + light `drop-shadow`, final perspective + rotateY + translate + skew + heavier `drop-shadow`.
  - **Slide** — initial `translateX(0%)`, final `translateX(±100%)`.
  - **Fade** — both frames identity, only opacity differs.
  - **Book Tilt** — initial `transform: none`, final perspective + rotateX + rotateY + scale.
- **New `OutgoingLayer` subcomponent** in `PageFlip.tsx` owns the phase state of the outgoing leaf:
  - `useState<'initial' | 'final'>('initial')` on mount.
  - Inside a single `useEffect`, schedule **two nested `requestAnimationFrame`s** before calling `setPhase('final')`. The double-RAF guarantees the browser commits the initial paint before the property change, so the transition has a real "from" frame to interpolate from.
  - The effect cleans up both RAFs and short-circuits the state update if the component unmounts (rapid double flips).
  - The layer exposes the current phase via `data-flip-phase`, giving tests and DevTools a hook to assert against.
- **`PageFlip` body** delegates the outgoing leaf entirely to `OutgoingLayer` (which handles both single-mode `<img>` and spread-mode wrapper variants). The existing `PageSurface` is now strictly for the static page.

## Trade-offs

- **Double RAF, not single**: a single `requestAnimationFrame` is enough on most browsers, but Safari and certain WebKit-based environments occasionally collapse the property change into the same paint as the mount. Double RAF costs ~16 ms of perceptual latency before the flip starts (already imperceptible to humans) and is reliable across every engine.
- **Phase state lives inside `OutgoingLayer`**, not lifted to `PageFlip`: keeps the parent component clean and means the layer is fully self-contained — instantiating `<OutgoingLayer />` anywhere will Just Animate.
- **Initial frame has the same `transition` string as final**: technically the initial frame doesn't need a transition (since it's the "from" state), but keeping the string identical avoids React swapping different `style` objects between phases and possibly retriggering layout.

### Regression coverage

- **Unit** (`PageFlip.test.tsx`):
  - `mounts the outgoing leaf in the resting "initial" frame, then advances to "final" on the next RAF` — asserts `data-flip-phase` and inline style cycle.
  - `animates the classic preset from no transform to rotateY in the final frame` — proves rotateY only appears after RAF.
  - `applies the fade preset opacity-only transition to the outgoing leaf` — initial opacity 1 → final 0.
- **E2E** (`e2e/reader-flip-presets.spec.ts`):
  - `outgoing leaf advances from the resting frame to the preset final frame` — clicks Next, asserts `data-flip-phase=final` and inline `transform: translateX(-100%)` for the Slide preset.
  - `outgoing leaf actually interpolates its transform across multiple frames (no instant snap)` — samples `getComputedStyle().transform` over ~350 ms of a 700 ms transition and asserts ≥ 2 distinct values, proving real animation frames are being painted.

## Follow-ups

- Stop using inline-style transitions altogether — switch to CSS keyframes via `Element.animate()` for richer easing curves and the ability to chain phases. Tracked separately if/when we add a sixth preset.
- The reduced-motion fallback could share a "soft" final transform instead of pure opacity, once design defines what "respectful but not invisible" should look like.

# Issue #8 — feat(reader): implement css 3d page-flip animation

- **Status**: Resolved
- **Milestone**: M2 — Reader MVP
- **Branch**: `feat/8-css-3d-page-flip`

## Problem

The reader showed a static page placeholder. The product needs the actual page-flip animation — the central interaction of `book-flip-showcase`.

## Approach

- Added `src/components/reader/PageFlip.tsx`:
  - Controlled component (`pages`, `pageIndex`, `onPageChange`, `ariaLabel`, `flipDurationMs`).
  - Renders the current page inside a 2400px-perspective container with a spine highlight and edge shadow gradients to suggest a book spread.
  - When `pageIndex` changes, captures the previous page as an "outgoing" leaf and rotates it 180° around the left or right edge (forward / backward) while fading. After `flipDurationMs`, the outgoing leaf is dropped, leaving the new current page.
  - Click on the right half advances; click on the left half rewinds. Enter / Space advances. Clicks while animating are ignored to prevent jumps.
  - Preloads the `pageIndex ± 1` images via the `Image` constructor for no-jank flips.
- Added `src/hooks/useReducedMotion.ts`: SSR-safe hook around `window.matchMedia('(prefers-reduced-motion: reduce)')` with live updates.
- `PageFlip` swaps its rotate/transform animation for an opacity cross-fade when reduced motion is preferred, and clamps the duration to ≤220ms.
- Wired `Reader.tsx` to own `pageIndex` state and pass it to `PageFlip`. Updated the footer copy to point users at the new click-to-flip gesture.
- Updated `Reader.test.tsx` so the page indicator and alt-text assertions match the new `Page N / Total` layout.

## Trade-offs

- **Single rotating leaf vs full spread**: keeps the DOM small and the math simple. A two-page spread (with center spine + facing pages) is visually closer to a real book but doubles the asset and layout cost; deferring until M3 / M4 polish.
- **Click zones in this PR**: gives the component an interactive surface so unit tests can verify boundary behavior without waiting for the dedicated controls in #9. The click hit-area will coexist with the prev/next buttons added by #9.
- **`Image()` preloading vs `<link rel=preload>`**: the constructor approach works inside React effects without touching `<head>`. Lighthouse perf work in #15 will revisit this if needed.

# Issue #55 — release-please auto-tagging on `release` branch

## Problem

The project deploys to GitHub Pages from the `release` branch (`.github/workflows/deploy.yml`), but version bumps, git tags, and GitHub Releases were still manual — the v1.0.0 tag and release were cut by hand in #18. Every subsequent merge from `main` into `release` therefore shipped code without a semver trail.

## Approach

Added [release-please](https://github.com/googleapis/release-please) in **manifest mode** so the current `1.0.0` baseline is bootstrapped and future bumps follow Conventional Commits.

### Files

| File | Role |
|---|---|
| `.github/release-please-config.json` | Node release strategy, `CHANGELOG.md` path, `v`-prefixed tags |
| `.github/.release-please-manifest.json` | Bootstrap version `1.0.0` (matches existing tag + `package.json`) |
| `.github/workflows/release-please.yml` | Runs on push to `release` (+ `workflow_dispatch`) |

### Workflow behaviour

1. Developer merges `main` → `release` (same trigger as Pages deploy).
2. `release-please-action@v4` scans conventional commits since the last tag (`v1.0.0`).
3. Semver is computed (`feat:` → minor, `fix:` → patch, `!` → major).
4. With `skip-github-pull-request: true`, release-please **tags and publishes a GitHub Release immediately** — no intermediate Release PR. It also commits the bumped `package.json`, `CHANGELOG.md`, and manifest back to `release`.
5. `deploy.yml` continues to run independently on the same push and publishes the built site.

### Release flow (operator)

```text
main  ──merge──▶  release  ──push──▶  release-please (tag + changelog commit)
                              └──▶  deploy.yml (GitHub Pages)
```

After a release, consider merging `release` → `main` so the version bump commit lands on the default branch.

## Trade-offs

- **Direct release vs Release PR**: We chose `skip-github-pull-request: true` so a single merge into `release` produces the tag — matching the request for automatic tagging on branch merge. The trade-off is no human review step for the generated changelog; acceptable for a solo/small-team repo with squash-merged PRs already reviewed on `main`.
- **Double workflow runs**: If release-please pushes a version-bump commit back to `release`, both workflows fire again; the second release-please run finds nothing new to release and is a no-op.

# Issue #11 — feat(reader): add keyboard shortcuts

- **Status**: Resolved
- **Milestone**: M3 — Advanced Navigation
- **Branch**: `feat/11-keyboard-shortcuts`

## Problem

The reader had no keyboard navigation. Power users and assistive tech users need at minimum ArrowLeft / ArrowRight, Home / End, and Escape to navigate without reaching for a mouse.

## Approach

- Added `src/hooks/useReaderKeyboard.ts`:
  - Attaches a single `keydown` listener on `document` while the hook is mounted.
  - Maps `ArrowLeft` / `ArrowRight` to prev / next, `Home` / `End` to first / last, `Escape` to a caller-supplied `onExit`.
  - Calls `event.preventDefault()` only when an action will fire, so disabled boundaries (ArrowLeft at page 1, ArrowRight at last page) and unhandled keys still propagate normally.
  - Skips when the event target is an `<input>`, `<textarea>`, `<select>`, or an element with `contenteditable`. Detects contenteditable both via `isContentEditable` and the raw attribute so jsdom (which does not implement the property) behaves the same as real browsers.
  - Ignores events with `metaKey` / `ctrlKey` / `altKey` so OS shortcuts (Cmd+Right, Ctrl+End in form fields, etc.) are not hijacked.
- Wired the hook into `Reader.tsx` with `onPageChange: setPageIndex` and `onExit: () => navigate('/')`.

## Trade-offs

- **`document.addEventListener` vs container-scoped**: a global listener is simpler than focus-trap plumbing. The form-field guard ensures the global listener does not interfere with the page-jump input (#10). The cost is a tiny global cleanup discipline; the cleanup runs on unmount.
- **`Escape` returns to gallery vs closes a modal**: there is no reader-level modal today, so `Escape = exit` is the highest-value mapping. When a modal lands (e.g., a settings sheet in M4), the modal will listen first and stop propagation.
- **Boundary behavior**: at edges we let the key bubble (no `preventDefault`) so scrolling and other defaults still work for users at the start / end of a book.

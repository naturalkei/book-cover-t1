# Issue #10 — feat(reader): add page-jump input

- **Status**: Resolved
- **Milestone**: M3 — Advanced Navigation
- **Branch**: `feat/10-page-jump-input`

## Problem

Long books were tedious to navigate with prev/next alone. The user needs to type a target page and land there immediately, with the flip animation still running for context.

## Approach

- Added `src/components/reader/PageJumpInput.tsx`:
  - A small `<form role="search" aria-label="Jump to page">` containing a labelled number `<input>` and a visually hidden submit button.
  - Maintains a local `draft` string so the user can type freely, with a synced "lastPageIndex" cell that resets the draft when `pageIndex` changes externally (no `useEffect`, matching the new `react-hooks/set-state-in-effect` rule).
  - `commit()` parses the draft, returns to the previous value on NaN, clamps to `[1, totalPages]`, updates the draft, and only fires `onPageChange` when the resulting index differs from the current one.
  - Commits on Enter (form submit) and on blur, so click-away also lands the jump.
  - Form opts out of HTML5 validation (`noValidate`) so our clamping logic always runs even when the typed value is out of range.
- Wired the input into `Reader.tsx`, centered under `ReaderControls`. Shares the same `setPageIndex` callback so the flip animation kicks in.

## Trade-offs

- **Local draft vs controlled-from-parent**: a local draft lets the user type interim values like `1` on the way to `12` without the parent re-rendering. Parent only sees the committed value.
- **`noValidate`**: drops the native popup for out-of-range values, but our own logic clamps and updates the visible value, which is the better UX for a single-field jump control.
- **No keyboard arrow-key handling**: number inputs already step on ArrowUp / ArrowDown, but that triggers `onChange`, not `commit`. Users still need Enter or blur to apply. Acceptable for now; M3's keyboard shortcut issue (#11) covers global arrow navigation on the reader instead.

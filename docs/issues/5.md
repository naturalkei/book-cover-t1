# Issue #5 — feat(data): declare static book metadata module

- **Status**: Resolved
- **Milestone**: M1 — Foundation
- **Branch**: `feat/5-book-metadata-module`

## Problem

The gallery (#6) and reader (#7+) need a single typed source of book metadata. There was no module owning that data.

## Approach

- Added `src/data/books.ts` exporting:
  - `Book` interface (`id`, `title`, `author`, `coverSrc`, `pages: string[]`, optional `accentColor`, optional `description`).
  - `books: Book[]` array with three sample books (Atlas of Cities, Silent Frequencies, Coastal Tales).
  - `getBookById(id: string): Book | undefined` helper.
- Created four bespoke SVG placeholders under `public/books/`:
  - One unique cover per book, each with a distinct palette/typography to make the gallery visually diverse out of the gate.
  - One shared `page-blank.svg` page placeholder referenced by every book's `pages` array. The reader will overlay page numbers in M2, so identical assets are sufficient now.
- Added `src/data/books.test.ts` covering: id uniqueness, non-empty pages, `getBookById` happy path, miss path, and case sensitivity.
- Deleted the temporary `src/test/sample.test.ts` from issue #3 now that real specs exist.

## Trade-offs

- **SVG placeholders vs external CDN**: SVGs keep the showcase fully offline, ship as tiny files, and stay deterministic in tests. CDN images would look richer but introduce flakiness in CI and require allowlisting domains.
- **Shared page placeholder vs unique pages**: One shared `page-blank.svg` keeps the repo lean; per-book page art can land in M5 alongside the demo media work.
- **`accentColor` typed loose as `string`**: keeping it as a free-form CSS color string for now; can tighten to a token set when theming lands (M4).

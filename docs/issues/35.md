# Issue #35 — feat(reader): single-page and spread (two-page) view modes

- **Status**: Resolved
- **Milestone**: M6 — Reader Enhancements
- **Branch**: `feat/35-view-modes`

## Problem

The reader always rendered one page in the flip surface. Showcasing a book-flip animation without a real spread misses the most natural form factor — two facing pages joined at the gutter — and ignores the extra horizontal real estate on desktop viewports. We needed a user-controllable view mode that is persistent, viewport-aware, and consistent across every navigation surface (controls, keyboard, page-jump, scrubber).

## Approach

- **`useViewMode` hook** (`src/hooks/useViewMode.ts`):
  - State: `'single' | 'spread'`, persisted under `book-flip-showcase:view-mode`.
  - Initial value resolution: stored preference → `matchMedia('(min-width: 1024px)')` → `'single'`. This means desktop visitors see the spread by default, mobile / narrow visitors get single.
  - Exports `setViewMode`, `toggleViewMode`, plus pure helpers `stepForMode(mode)` (1 vs 2) and `snapToStep(pageIndex, step)` (always snaps down to the nearest valid step boundary).
- **`ViewModeToggle` component** — a small radio-group segmented control with `Single` and `Spread` chips. Lives in the reader header next to the title/back link.
- **`PageFlip` extension**:
  - New `mode?: ViewMode` prop.
  - Aspect ratio swaps: `aspect-[3/4] max-w-2xl` for single, `aspect-[3/2] max-w-4xl` for spread.
  - In spread mode the surface renders two `<img>` elements side by side. The left page keeps the `page-flip-current` test id (so the perf assertion in `e2e/reader-perf.spec.ts` continues to work); the right page gets `page-flip-current-right`; the wrapper gets `page-flip-current-spread`. The outgoing layer mirrors the same structure.
  - A single private `PageSurface` helper renders both static and outgoing layers in either mode so the transform / transition logic lives in exactly one place.
  - Step semantics: click on the right half advances by 2 in spread; left half retreats by 2. Enter key forwards by step.
  - Preload widens to ±4 in spread (so the next/prev spreads are warmed), ±2 in single.
- **`ReaderControls`**:
  - New optional `step` prop. Prev / Next now move by `step`, and the indicator renders a range (`3–4 / 8`) when step > 1 and a trailing page exists. A new `reader-current-page-right` test id covers the right page of the range so the range is automatically asserted.
- **`useReaderKeyboard`** accepts `step`. Arrow keys move by step, `End` lands on the last spread start (`lastIndex − lastIndex % step`), `Home` still resets to 0, `Escape` still bubbles to the exit handler. Edge guards keep us inside the book.
- **`PageJumpInput`** accepts `step`. The committed target is snapped down to the nearest spread start; the displayed value is the snapped value so users see exactly where the reader will land. Form `noValidate` was kept so the custom clamp keeps owning the input.
- **`Reader` page** wires everything together:
  - Reads `viewMode` from the hook.
  - Holds `pageIndex` but always commits through `commitPage(next)` → `snapToStep(clampedNext, step)`. The same callback is passed to every consumer (controls, page-jump, scrubber, keyboard, flip), so any change funnels through one snap point.
  - Snaps `pageIndex` whenever the step changes (e.g. user switches from single → spread mid-read so they never end up on an invalid odd index).
- **E2E**:
  - New `e2e/reader-view-modes.spec.ts`: default mode on narrow viewports, switching to spread side-by-side renders, range indicator, persistence across reload, keyboard step.
  - New helper module `e2e/helpers.ts` (seedTheme, seedViewMode, openFirstBook).
  - Existing reader specs (`reader-navigation`, `reader-keyboard`, `reader-scrubber`, `reader-perf`, the a11y reader-keyboard and reduced-motion specs) now seed `single` mode so their step-by-1 contracts stay deterministic regardless of the default viewport's preferred mode.
- **Unit / component tests**:
  - `useViewMode`: persistence, breakpoint detection, invalid-value handling, `toggleViewMode`, `stepForMode`, `snapToStep`.
  - `ViewModeToggle`: rendering, active state, click handler, radiogroup semantics.
  - `PageFlip`: spread renders both pages, right-half click advances by 2, left-half retreats by 2, empty trailing slot for odd totals.
  - `ReaderControls`: spread range display, step-by-2 next/prev, range hidden on solo last page.
  - `useReaderKeyboard`: step-by-2 arrow keys, End lands on last spread start (even and odd totals), refuses past last spread.
  - `PageJumpInput`: spread snaps an odd target down to the nearest spread start and suppresses a no-op when the snapped target equals the current index.

## Trade-offs

- **Single set of test ids**: I deliberately reused `page-flip-current` for the left page in spread mode so existing perf tests keep their semantics. The wrapper gets a distinct `…-spread` id when callers need to assert on the container. This avoided an across-the-board refactor of unrelated specs.
- **Snap down, not nearest**: the page-jump input could have rounded to the nearest spread start, but rounding down keeps the user's mental model simple ("the page I asked for will be visible") and avoids backward jumps on near-boundary input.
- **Default mode is viewport-aware**: deliberately a heuristic, not a strict media query. Once the user picks a mode the choice persists permanently. The only downside is that an existing user resizing across the breakpoint won't see the layout swap automatically; this matches the way browsers handle reader-mode preferences.
- **Animation in spread mode**: for v1 the outgoing layer animates the whole spread as a unit (same rotateY transform as single). A truer "only the right page flips" effect needs a slightly different layer composition and is intentionally deferred to #36 (animation presets).

## Follow-ups

- #36 (animation presets) should add a "two-page-aware" preset where only the relevant half flips, taking advantage of the wrapper / inner-image split landing in this PR.
- Consider sliding the toggle into the `ReaderControls` row on very narrow viewports so it doesn't push the header height.
- Consider a "auto" mode that re-snaps on viewport change instead of only on first paint.

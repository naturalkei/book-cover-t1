# Issue #17 — docs: add demo screenshots and gif to readme

- **Status**: Resolved
- **Milestone**: M5 — Release
- **Branch**: `docs/17-screenshots-readme`

## Problem

The README opened with badges and prose only. For a showcase project, the first scroll needs to look like a showcase — a hero image and side-by-side light/dark captures so the reader can judge the product at a glance.

## Approach

- **New media folder**: `docs/media/` with four PNG screenshots (`hero.png`, `gallery-light.png`, `reader-dark.png`, `reader-light.png`) plus a small README documenting provenance.
- **Reproducible capture**: `scripts/screenshots.ts` uses Playwright's bundled Chromium at `1440×900` with `deviceScaleFactor: 2`. Each shot:
  - Seeds `localStorage` with the desired theme before navigation.
  - Drives the app the same way a user would (click the first gallery card to enter the reader, advance one page so the spread is visible).
  - Saves into `docs/media/`.
- **New script**: `pnpm screenshots` → `tsx scripts/screenshots.ts`. Added `tsx` as a dev dependency so the TS script runs without a pre-build step.
- **README pass**:
  - Inserted the hero image right under the tagline.
  - Added a "Screenshots" section with two 2×1 tables (dark / light × gallery / reader).
  - Updated the scripts table to include the new `test`, `test:watch`, `e2e`, `e2e:ui`, `screenshots` entries.
  - Refreshed the "Project Structure" tree to mention `e2e/`, `docs/media/`, `scripts/`, `src/hooks/`, lazy-loaded Reader, etc.
  - Bumped the "Unit Tests / E2E Tests" rows in the tech stack table from "planned" to the real versions.

## Trade-offs

- **Static PNGs over an MP4/GIF**: a real animated demo would be more compelling, but recording video reliably from a sandboxed Playwright runner is brittle and the file size grows quickly. PNG screenshots keep the repo lean, render in GitHub instantly, and stay reproducible from the same script we just wrote. If a GIF becomes a hard requirement later, the script can swap `page.screenshot` for `page.video()` without restructuring.
- **Reader frame is past page 1**: every reader capture advances one page first, so the screenshot includes both the spread and the active state of the controls (Prev enabled, current = 2). Trade-off: the cover-as-page-1 illustration is hidden in those shots. Worth it because the controls are the actual differentiator.
- **No image optimization step**: PNGs are ~200–550 KB each. Cleanly under GitHub's render budget and they only need to load on the README. If they ever balloon, run them through `oxipng` or convert to WebP.

## Verification

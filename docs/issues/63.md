# Issue #63 — Variable naming conventions and clsx Tailwind grouping

## Problem

Type/interface/constant naming was inconsistent (`Book`, `CoverMode`, `FLIP_PRESET_LIST`), and long Tailwind `className` strings hurt readability. No ESLint or docs captured the intended conventions.

## Approach

1. **Documented conventions** in `docs/plan-1.md` §4 and `.cursor/rules/code-style.mdc`:
   - `interface` → `IPascalCase`
   - `type` → `TPascalCase`
   - Literal constants → `UPPER_SNAKE_CASE`
   - Constant objects/arrays → `PascalCase`
   - Functions → camelCase with verb prefix (React components remain PascalCase)

2. **ESLint** — `@typescript-eslint/naming-convention` on `src/**/*` (interface `I*`, typeAlias `T*`, functions camelCase/PascalCase).

3. **Renamed symbols** across `src/` — e.g. `Book` → `IBook`, `books` → `Books`, `CoverMode` → `TCoverMode`, `FLIP_PRESET_LIST` → `FlipPresetList`, `stepForMode` → `getStepForMode`, `effectiveStep` → `getEffectiveStep`.

4. **Tailwind** — added `clsx` dependency and `src/lib/class-names.ts` for shared focus-ring / button bundles; refactored components with 80+ char class strings to multi-line `clsx(...)`.

## Trade-offs

- Shared class bundles in `class-names.ts` reduce duplication but add an indirection layer — acceptable for repeated focus-ring patterns.
- React component functions stay PascalCase (ESLint allows both camelCase and PascalCase for functions).
- Historical `docs/issues/*.md` paths unchanged.

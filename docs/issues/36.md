# Issue #36 — feat(reader): selectable page-flip animation presets

- **Status**: Resolved
- **Milestone**: M6 — Reader Enhancements
- **Branch**: `feat/36-flip-presets`

## Problem

The reader shipped exactly one outgoing-leaf animation: a 3D `rotateY(180deg)` flip plus a reduced-motion opacity fallback baked into `PageFlip`. The flip is the marquee feature of this project, so users (and our future selves) should be able to compare presets live without forking the component. The 5-preset target also forces the animation system to become extensible instead of hardcoded.

## Approach

- **Preset registry** (`src/components/reader/flipPresets.ts`):
  - `FlipPreset = { id, label, description, build(direction, durationMs) => CSSProperties }`.
  - Five presets:
    1. **Classic** — `rotateY(±180deg)` with the existing `cubic-bezier(0.2, 0.7, 0.2, 1)` curve. Default.
    2. **Curl** — `rotateY(±160deg)` + `translateX(±6%)` + `skewY(±1deg)` with a `drop-shadow` filter for a paper-lift feel.
    3. **Slide** — flat `translateX(±100%)`, no rotation.
    4. **Fade** — `opacity 0` only, used as the reduced-motion fallback.
    5. **Book Tilt** — `rotateX(-10deg)` + `rotateY(±150deg)` + `scale(0.92)`.
  - Pure module: no React, no DOM, fully unit-testable.
  - Exports `DEFAULT_FLIP_PRESET = 'classic'`, `REDUCED_MOTION_PRESET = 'fade'`, `getFlipPreset(id)`, `isFlipPresetId(value)`.
- **`useFlipPreset` hook** (`src/hooks/useFlipPreset.ts`):
  - Persists selection under `book-flip-showcase:flip-preset`.
  - Subscribes to `useReducedMotion`; when the user prefers reduce, returns `effectivePreset = 'fade'` regardless of the stored preference but keeps `preset` as the stored value so the UI can show "you picked Tilt, fade is in effect".
- **`FlipPresetPicker` component**:
  - Renders one chip per registered preset under a `role="radiogroup"` with an accessible label.
  - Each chip carries `aria-checked` for the stored value and `data-active` for the effective value, so a screen reader announces the user's choice even when reduced motion is forcing fade.
  - A "reduced motion · fade only" caption renders when `locked` is true.
- **`PageFlip` refactor**:
  - Accepts `presetId`.
  - Outgoing leaf style is now produced by `getFlipPreset(presetId).build(direction, duration)` — the inline rotateY/transition fork is gone.
  - The static page still gets the same reduced-motion dim (`opacity 0.4` while outgoing exists).
  - Surface exposes the active preset via `data-flip-preset` so both unit and E2E tests can assert without DOM mutation inspection.
- **`Reader` page** wires `useFlipPreset` and passes `effectivePreset` to `PageFlip` while feeding `preset` + `effectiveValue` to the picker so the UI stays honest.
- **Tests**:
  - `flipPresets.test.ts` — registry shape (count, ids, default/fallback, labels), per-preset `build()` returns valid CSS, fade is opacity-only, classic uses mirrored rotateY origins, `isFlipPresetId` guards.
  - `useFlipPreset.test.tsx` — default, restore, invalid-value rejection, persistence, reduced-motion override of `effectivePreset` (stored value preserved).
  - `FlipPresetPicker.test.tsx` — chip rendering, aria-checked vs data-active distinction, locked caption.
  - `PageFlip.test.tsx` — `data-flip-preset` attribute, slide and fade outgoing styles applied (transform / no-rotate respectively).
  - `e2e/reader-flip-presets.spec.ts` — default surfaces classic, switching swaps + persists across reload, outgoing leaf transform differs by preset, reduced motion forces fade and surfaces the locked caption.

## Trade-offs

- **Presets are CSS-only**: no per-preset JS step, no canvas/WebGL. Keeps the bundle tiny (Reader chunk only grew from 3.99 → 6.34 KB gzipped) and makes adding a sixth preset literally a one-record append.
- **Same flip for both view modes**: in spread mode the outgoing element is the entire spread, animated as one card. A truly "only the right page flips" effect needs a different layer composition; deliberately deferred.
- **Reduced motion overrides stored preference**: I prefer the user's accessibility setting always wins over an explicit choice. The picker still highlights the user's stored chip so they can see what would be active without the override.
- **Picker placement**: lives under the scrubber, not in the header, because it's a tertiary control. Could move into a popover or a settings drawer if it ever crowds the layout.

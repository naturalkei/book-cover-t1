# Issue #3 — test: scaffold vitest and react testing library

- **Status**: Resolved
- **Milestone**: M1 — Foundation
- **Branch**: `test/3-scaffold-vitest`

## Problem

The project had no unit / component testing infrastructure. The `test` npm script printed an error and exited non-zero, blocking the agent workflow's mandatory Vitest step.

## Approach

- Installed `vitest`, `@vitest/coverage-v8`, `@testing-library/react`, `@testing-library/user-event`, `@testing-library/jest-dom`, `@testing-library/dom`, `jsdom`.
- Added `vitest.config.ts` reusing the existing `@vitejs/plugin-react` and `vite-tsconfig-paths` so the `@/*` alias and React JSX runtime work in tests.
- Enabled `globals: true`, `environment: 'jsdom'`, and pointed `setupFiles` at `src/test/setup.ts`.
- Wrote `src/test/setup.ts` that imports `@testing-library/jest-dom/vitest` (registers DOM matchers on Vitest's `expect`) and runs `cleanup()` after each test.
- Replaced the placeholder `test` script with `vitest run`; added `test:watch` and `test:coverage`.
- Added `vitest.config.ts` to `tsconfig.node.json` `include` so it is typechecked under the node project, not the browser app project.
- Wrote a minimal `src/test/sample.test.ts` to prove the wiring (two passing assertions, one of which uses a jest-dom matcher).

## Trade-offs

- Kept `globals: true` for ergonomic test authoring even though the sample spec imports explicitly. Consumers may switch to explicit-only later if they prefer.
- Did not configure a coverage threshold — that belongs in a later quality-gate issue once real source coverage exists.
- Sample spec lives under `src/test/` (not colocated) so it can be deleted in a follow-up without touching real component tests.

## Verification

- `pnpm test` → `Test Files 1 passed (1), Tests 2 passed (2)`.
- `pnpm lint` → clean.

## Follow-ups

- Delete `src/test/sample.test.ts` when the first real component spec lands (issue #5 or #6).
- Decide on coverage thresholds once the source tree stabilizes.

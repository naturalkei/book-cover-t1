# Issue #45 — feat(reader): single-leaf flip animation in spread mode (real-book turn)

- **Status**: Resolved
- **Milestone**: M6 — Reader Enhancements
- **Branch**: `feat/45-spread-single-leaf-flip`

## Why

Previously a forward (or backward) flip in spread mode rotated **both** facing pages as one rigid surface. A real book turns a **single leaf** at a time: the right leaf pivots around the spine, its back face becomes the new left page, and the new right page is uncovered behind it. This issue brings that physical model to the reader.

## How

### 1. `FlipPreset.build()` is now mode-aware

`build(direction, duration, mode)` accepts a `ViewMode`. In **`spread`** mode:

| Preset  | Final transform (spread)                                                      | Notes                                                  |
| ------- | ----------------------------------------------------------------------------- | ------------------------------------------------------ |
| classic | `rotateY(±180deg)`                                                            | Spine pivot. Opacity stays 1 so the back face shows.   |
| curl    | `perspective(1600px) rotateY(±178deg)`                                        | Tighter rotation; opacity 1.                           |
| slide   | `translateX(±110%)`                                                           | Half-leaf slides _away from_ the spine; opacity → 0.   |
| fade    | unchanged                                                                     | Opacity-only, both faces fade together.                |
| tilt    | `perspective(1800px) rotateX(-6deg) rotateY(±178deg) scale(0.96)`             | Subtler tilt to keep the leaf legible at near-180°.    |

`backfaceVisibility: hidden` stays on every **single-mode** frame (no double-painted back). In **spread mode** the property is removed from the frame so it can be applied per-face on the two-sided leaf inside `OutgoingSpreadLeaf`.

### 2. New `OutgoingSpreadLeaf` component

When `mode === 'spread'` and neither the outgoing nor the current state is the cover-alone page, the outgoing rendering switches from the full-surface `OutgoingLayer` to a new two-element subtree:

- **`page-flip-phantom`** — a half-width element on the **opposite** side from the flipping leaf, holding the **old** content of that side. It stays visible underneath until the leaf's back face physically lands on top.
- **`page-flip-outgoing`** — a half-width element on the flipping side with `transformStyle: 'preserve-3d'`, containing two `<img>` faces:
  - **front** (`backfaceVisibility: hidden`) — the old content of the flipping side.
  - **back** (`backfaceVisibility: hidden` + `transform: rotateY(180deg)`) — the new content of the **opposite** side.

The container animates per the preset; the spine is the pivot (`left center` forward, `right center` backward); a double-`requestAnimationFrame` triggers the phase swap so the from-frame is painted (regression test from #39 still applies).

### 3. Direction → side mapping

For a forward step from old spread `[L_o, R_o]` to new spread `[L_n, R_n]`:

- Flipping leaf on the **right** half.
- Front face = `R_o`, back face = `L_n`.
- Phantom on the **left** half = `L_o`.

For a backward step from `[L_o, R_o]` to `[L_n, R_n]` (i.e. `pageIndex` decreasing):

- Flipping leaf on the **left** half.
- Front face = `L_o`, back face = `R_n`.
- Phantom on the **right** half = `R_o`.

This matches the physical book model — the leaf you flip carries the page you were just looking at on its front and the page that will replace its sibling on its back.

### 4. Static layer underneath

The static `PageSurface` continues to render the **new** spread immediately. Combined with the phantom + leaf, the user sees:

| Phase            | Left side                                  | Right side                                |
| ---------------- | ------------------------------------------ | ----------------------------------------- |
| 0°               | Phantom `L_o` (covering static `L_n`)      | Leaf front `R_o` (covering static `R_n`)  |
| 90°              | Phantom `L_o`                              | Static `R_n` (leaf edge-on)               |
| 180°             | Leaf back `L_n` (covering phantom `L_o`)   | Static `R_n`                              |
| post-cleanup     | Static `L_n`                               | Static `R_n`                              |

The transition from "leaf back `L_n` on top" to "static `L_n` underneath" is content-identical, so the cleanup is invisible.

### 5. Cover transitions stay on the original path

Cover ↔ spread transitions use the original `OutgoingLayer` (which renders the whole surface) because the cover surface is asymmetric and the leaf-pair semantics don't apply cleanly. The `shouldUseSpreadLeaf()` predicate gates this — if either the outgoing or the current index is a cover-alone page, the legacy path is used.

- [x] All five presets keep working; rotational ones reveal the back face mid-rotation; non-rotational ones still apply only to the flipping side (the opposite side falls back to the static `L_n` / `R_n` at the end).

## Trade-offs

- **Phantom layer**: only the rotational presets actually use it (the back face arrives on top of it). Non-rotational presets (slide, fade) leave a hairline opportunity for the opposite side to snap from `L_o` to `L_n` at cleanup. The user's brief was "한쪽씩만" (one side at a time), so the opposite side is intentionally not animated.
- **Cover ↔ spread transitions** use the legacy whole-surface path. A two-sided leaf for those would need special-casing for the asymmetric (cover vs. inside) layout; left as a follow-up.
- **`transformStyle: 'preserve-3d'`** sits inline on the leaf wrapper so we can still set `boxShadow` / `filter` on the same node. Hoisting it to a Tailwind class would require a custom utility.

## Follow-ups

- Two-sided treatment for cover ↔ spread transitions.
- Drag-to-curl gesture (the leaf already understands the spine).
- Per-preset duration overrides (the global `flipDurationMs` is currently shared).

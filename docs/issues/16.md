# Issue #16 — ci: configure github pages deployment

- **Status**: Resolved
- **Milestone**: M5 — Release
- **Branch**: `ci/16-pages-deployment`

## Problem

The repository inherited a `deploy.yml` that triggered on a `release` branch, used a loose `pnpm install`, and ran without a peer CI workflow. To meet the project's Definition of Done (lint + unit + E2E suites green in CI) and ship to GitHub Pages from `main`, the workflows needed to be tightened and a CI workflow added.

## Approach

- **Updated `deploy.yml`**:
  - Trigger changed from `release` to `main` (plus `workflow_dispatch` for manual runs).
  - `concurrency: { group: pages, cancel-in-progress: true }` so successive pushes don't queue.
  - `pnpm install --frozen-lockfile` instead of plain `pnpm install`.
  - Pinned pnpm to v10 to match `packageManager` in `package.json`.
  - Existing `actions/upload-pages-artifact@v3` + `actions/deploy-pages@v4` deployment kept.
- **Added `ci.yml`** running on every PR and push to `main`:
  - One job, in order: `pnpm lint` → `pnpm test` → `pnpm build` → `pnpm e2e`.
  - Playwright browsers cached under `~/.cache/ms-playwright`, keyed on `pnpm-lock.yaml`. System deps installed conditionally so cache hits stay fast.
  - Uploads the Playwright HTML report as an artifact (7-day retention).
  - Concurrency group keyed on `(workflow, ref)` so PR pushes cancel stale runs.
- **README**: added the CI and Deploy workflow status badges at the top, replaced the bare "Deployment" section with a fuller "Continuous Integration & Deployment" description covering both workflows.

## Trade-offs

- **Single `verify` job vs. matrix**: kept everything in one job because the steps share the same Node / pnpm setup and the e2e job depends on the build. Splitting into parallel jobs would shave a minute but quadruple the setup overhead per run. Easy to revisit if total runtime ever exceeds ~10 minutes.
- **Chromium-only Playwright**: matches `playwright.config.ts` (only Chromium project). Adding Firefox / WebKit later is a config change plus a cache-key bump, not a workflow rewrite.
- **No deploy-on-PR previews**: GitHub Pages doesn't support deploy previews without extra infrastructure. Out of scope for v1.0.0; can switch to Cloudflare Pages or similar if previews become a hard requirement.

## Verification

- `pnpm lint` → clean locally.
- `pnpm test` → 69 passing.
- `pnpm e2e` → 16 passing.
- `pnpm build` → bundle within the 200 KB gzipped budget (77.98 KB initial + 3.99 KB Reader chunk).
- Workflow YAML structure validated by GitHub Actions's parser (`gh workflow list` shows both).

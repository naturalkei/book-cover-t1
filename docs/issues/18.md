# Issue #18 — chore(release): cut v1.0.0 tag

- **Status**: Resolved
- **Milestone**: M5 — Release
- **Branch**: `chore/18-release-v1`

## Problem

M1–M4 plus the docs and CI tracks under M5 are merged to `main`. We needed a single audit trail of what shipped, an annotated `v1.0.0` tag, and a GitHub Release with notes mirroring the changelog so the deployed site has a versioned moment in time to point at.

## Approach

- **`CHANGELOG.md`** at the repo root, grouped by milestone (Foundation / Reader MVP / Advanced Navigation / Polish / Release) and linked back to every closed issue and the upcoming tag. Notes call out:
  - The 77.98 kB gzipped initial-JS bundle and the route-split reader chunk.
  - The 69 Vitest specs and 16 Playwright specs the release is built against.
  - The `prefers-reduced-motion`, axe-clean, and theme-toggle stories.
- **`package.json` `version`** kept at `1.0.0` (it was set when scaffolding the project, so no bump is needed — the tag and release simply ratify it).
- **Annotated tag + release**: pushed `v1.0.0` with a multi-line annotated message, then opened a GitHub Release whose body mirrors the changelog and embeds the hero screenshot.

## Trade-offs

- **No `package.json` patch bump**: the manifest already declared `1.0.0`. Bumping to `1.0.1` here would have produced a tag/version mismatch and added churn. Future patch releases will follow SemVer normally.
- **Changelog grouped by milestone, not commit type**: easier to scan and matches the way the plan and the issue board are laid out. Conventional Commit prefixes are still preserved in each entry's prose.
- **Release notes mirror the changelog**: keeps a single source of truth so we never have to choose which copy to trust.

## Verification

- `pnpm lint` → clean.
- `pnpm test` → **69 / 69**.
- `pnpm e2e` → **16 / 16**.
- `pnpm build` → bundle within the 200 kB gzipped budget.
- Tag verified with `git show v1.0.0`; release verified with `gh release view v1.0.0`.

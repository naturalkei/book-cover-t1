## Problem

The app shipped a single dark theme. The product needed an explicit light / dark toggle, persisted across reloads, and respectful of the user's OS-level `prefers-color-scheme` when no preference had been stored.

## Approach

- Switched Tailwind to the **class-based dark mode** strategy. Added
  ```css
  @custom-variant dark (&:where(.dark, .dark *));
  ```
  to `src/index.css` so every `dark:` utility now applies when `<html class="dark">`.
- Made the body baseline **light** (`bg-slate-50`) with a dark override under `html.dark`, and inverted the custom scrollbar so both themes look intentional.
- Added `src/hooks/useTheme.ts`:
  - Detects the initial theme via stored preference â†’ `prefers-color-scheme: light` â†’ fallback to dark.
  - On mount and on change, toggles the `.dark` class and `data-theme` attribute on `<html>` and persists the value to `localStorage` under `book-flip-showcase:theme`.
  - Exposes `theme`, `setTheme(value)`, `toggleTheme()`.
  - `applyInitialTheme()` is called from `main.tsx` before React mounts to avoid a flash of incorrect theme.
- Added `src/components/ThemeToggle.tsx`, a single-icon button (`Sun` for "go light", `Moon` for "go dark") wired to `toggleTheme`. Placed in the header next to the GitHub link.
- Refactored color classes across `Layout`, `Gallery`, `BookCard`, `NotFound`, `Reader`, `PageFlip`, `ReaderControls`, `PageJumpInput`, and `ThumbnailScrubber` so each surface has a sensible appearance in both themes (light = white / slate-50 / slate-200, dark = slate-900 / slate-950 / white/5).

## Trade-offs

- **Touching every component**: required because the design previously assumed a single palette. A token-driven approach (CSS vars) was considered, but Tailwind's dark variant is more idiomatic and keeps autocomplete intact.
- **Dark as the soft default**: when both `localStorage` and OS preference are silent, we land on dark â€” the brand-defining surface. We could have flipped this, but the marketing surfaces (gallery hero, covers) are designed dark-first.
- **`useEffect` on theme change vs imperative apply**: an effect cost one extra microtask per toggle, but it keeps storage writes and DOM writes co-located with the React lifecycle and avoids tearing if the hook is consumed by multiple components.

# #73 — v2 reader spine center shadow pop at `final`

## Problem

In spread mode with the classic preset, the center gutter shadow appeared to **pop** when the outgoing leaf reached `data-flip-phase="final"`. Manual repro: `/v2/book/:id` → spread view → flip forward/back with DevTools 6× slowdown — the spine darkening jumped at the phase boundary.

## Root cause

Two effects compounded at different times:

1. **Static spine overlay** (`page-flip-spine`) intensified immediately when `outgoing` mounted (`via-black/45 opacity-100` vs idle `via-black/25 opacity-80`, 300ms transition) — before the leaf animation started.
2. **At `final` phase** (double-RAF), spread classic/curl frames stepped `boxShadow` / `filter: drop-shadow` from rest to lifted values, adding a second dark band at the spine pivot while the overlay was already strong.

The visible “pop” aligned with the phase swap because leaf shadow began transitioning there, on top of an already-boosted spine overlay.

## Approach

1. **Phase-synced spine overlay** — track `outgoingPhase` from outgoing components; spine opacity ramps from `0.8` → `1.0` only when phase is `final`, using the same duration and `FLIP_EASE_PAPER` as the leaf.
2. **Constant spread leaf shadow** — classic spread `boxShadow` and curl spread `filter` are identical in `initial` and `final`; gutter depth comes from the overlay, not a phase-boundary shadow step.
3. **Unified spine gradient** — single `via-black/35` stop; no class toggle on `via-*` at phase boundaries.

v1 reader untouched.

## Trade-offs

- Spread classic/curl leaves no longer gain a heavier drop shadow mid-flip; motion reads slightly flatter but gutter shading is smoother.
- Single-mode shadow behavior unchanged (still rest → lifted).
- Spine overlay uses opacity only (not progress-driven CSS variables); sufficient for M2.2c scope.

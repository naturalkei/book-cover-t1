# Issue #9 — feat(reader): add prev/next navigation controls

- **Status**: Resolved
- **Milestone**: M2 — Reader MVP
- **Branch**: `feat/9-prev-next-controls`

## Problem

After #8, page flipping only worked via clicking the page surface. The product needs an obvious, accessible pair of buttons that drive the same animation, plus a visible page indicator.

## Approach

- Added `src/components/reader/ReaderControls.tsx`:
  - A `<nav aria-label="Reader pagination">` containing **Prev**, **Next**, and a live-updating page indicator (`N / Total`).
  - Buttons have explicit `aria-label`s, focus-visible rings, and disabled state at boundaries.
  - The indicator splits current / total into separate spans with `data-testid` hooks so Playwright and Vitest can target each value without relying on whitespace shape.
- Wired `Reader.tsx` to render `<ReaderControls />` under the `<PageFlip />`, sharing the same `setPageIndex` callback. The footer copy was simplified now that the indicator is part of the controls.
- Added `e2e/reader-navigation.spec.ts`: opens the first book from the gallery, asserts the prev button starts disabled, clicks Next through every page, asserts the next button ends disabled, then clicks Prev once.
- Updated `Reader.test.tsx` to read the page indicator from the `Reader pagination` nav.

## Trade-offs

- **Single component vs split**: kept Prev, Next, and the indicator in one `ReaderControls` so they share layout, ARIA grouping, and live-region semantics. Splitting would have multiplied props with no gain.
- **`data-testid` on numeric spans**: violates the "selectors first" guideline, but the test ids are the cleanest way to disambiguate the two digit nodes inside the same live region without changing visible text. Acceptable trade-off and documented in the spec.
- **Disabled vs `aria-disabled`**: chose native `disabled` since it gives correct semantics for assistive tech and prevents focus-on-click. Users can still tab past disabled buttons.

# Issue #15 — perf: lazy-load page images, preload neighbors, code-split reader

- **Status**: Resolved
- **Milestone**: M4 — Polish
- **Branch**: `perf/15-lazy-preload`

## Problem

The gallery and reader needed an explicit performance pass:

- Initial JS payload had to stay under the 200 KB gzipped budget.
- The current page should paint as fast as possible while neighbors are warmed up off the critical path.
- Off-screen images (book covers, scrubber thumbnails) should never block first paint.
- All of this had to be verifiable via automated tests so future regressions get caught.

## Approach

- **Route-level code split**: converted `Reader` into a `React.lazy()` import with a `<Suspense>` fallback in `src/App.tsx`. The gallery no longer pulls Reader's flip / scrubber / keyboard code into its initial chunk.
- **PageFlip image hints**: the current page `<img>` now sets `loading="eager"`, `decoding="async"`, and `fetchPriority="high"`. The outgoing animation image gets `decoding="async"` so the decode never blocks the flip animation.
- **Wider preload window**: `PageFlip` already preloaded ±1 neighbors via `new window.Image()`. Widened to a ±2 window so jumps to nearby pages feel instant on slower networks.
- **Below-the-fold images stay lazy**: `BookCard` and `ThumbnailScrubber` thumbnails already used `loading="lazy"` + `decoding="async"`; left those alone.
- **Tests**:
  - Updated `PageFlip` unit tests for the new ±2 preload window and asserted the high-priority attributes on the current page.
  - Added `e2e/reader-perf.spec.ts`:
    - Asserts the current page `<img>` carries `loading="eager"`, `decoding="async"`, `fetchpriority="high"`.
    - Asserts the Reader chunk is **not** requested when only the gallery is loaded, then is requested after navigating to a book.
    - Asserts only ≤ 2 DOM `<img>` elements live inside the flip surface at rest (preload uses the `Image` constructor, not the DOM).

## Trade-offs

- **Suspense fallback flash**: lazy-loading Reader trades a smaller gallery bundle for a brief loading state on first reader visit. Mitigated with a styled `role="status"` fallback; payload is tiny (~4 KB gzipped) so the flash is sub-frame on a fast network. Worth it.
- **Wider preload window**: ±2 means up to four extra image requests per page settle. For the current placeholder set those all dedupe to a single cached SVG; for real books this is the right trade. If a book ever gets very large pages, this can be tightened or made adaptive.
- **No Lighthouse gate in CI yet**: the issue's "Lighthouse ≥ 90" target is intentionally not enforced in code — that lives in #16 (CI workflow) and #17 (release artifacts) so we can attach an actual report. The bundle-size budget and the lazy / eager attribute assertions are the enforceable proxies for now.

## Verification

- `pnpm test` → **Test Files 11 passed (11), Tests 69 passed (69)** (one new spec).
- `pnpm e2e` → **16 passed** (3 new perf specs + 13 existing).
- `pnpm lint` → clean.
- `pnpm build` → bundle budget met:
  - `dist/assets/index-*.js`: **242.49 kB raw / 77.98 kB gzipped** (initial chunk, was 252.21 kB / 80.36 kB before splitting).
  - `dist/assets/Reader-*.js`: **11.45 kB raw / 3.99 kB gzipped** (lazy chunk).
  - `dist/assets/index-*.css`: 33.00 kB / 6.36 kB gzipped.
  - Total gzipped JS well under the 200 KB budget, with the gallery alone at ~78 KB.
